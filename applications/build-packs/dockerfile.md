Dockerfile содержит пошаговые инструкции по сборке Docker-образа, который Coolify использует для развертывания вашего приложения или веб-сайта.

Билдпак Dockerfile позволяет использовать ваш собственный Dockerfile для деплоя приложения, обеспечивая полный контроль над тем, как приложение собирается и запускается в Coolify.

## Как использовать Dockerfile?

### 1. Создайте новый ресурс в Coolify

В дашборде Coolify откройте свой проект и нажмите кнопку **Create New Resource**.

<ZoomableImage src="/docs/images/builds/packs/dockerfile/1.webp" alt="скриншот дашборда Coolify" />

### 2. Выберите вариант развертывания

<ZoomableImage src="/docs/images/builds/packs/dockerfile/2.webp" alt="скриншот дашборда Coolify" />

**A.** Если ваш Git-репозиторий публичный, выберите вариант **Public Repository**.

**B.** Если ваш репозиторий приватный, вы можете выбрать **Github App** или **Deploy Key**. (Эти методы требуют дополнительной настройки. Вы можете ознакомиться с руководствами по настройке [Github App](/applications/ci-cd/github/integration#with-github-app-recommended) или [Deploy Key](/applications/ci-cd/github/integration#with-deploy-keys).)

### 3. Выберите ваш Git-репозиторий

Если вы используете публичный репозиторий, вставьте URL вашего репозитория GitHub при появлении запроса. Шаги практически одинаковы для всех остальных вариантов.

<ZoomableImage src="/docs/images/builds/packs/dockerfile/3.webp" alt="скриншот дашборда Coolify" />

### 4. Выберите билдпак

Coolify по умолчанию предлагает использовать Nixpacks. Нажмите на опцию Nixpacks и выберите **Dockerfile** в качестве билдпака из выпадающего меню.

<ZoomableImage src="/docs/images/builds/packs/dockerfile/4.webp" alt="скриншот дашборда Coolify" />

### 5. Настройте билдпак

<ZoomableImage src="/docs/images/builds/packs/dockerfile/5.webp" alt="скриншот дашборда Coolify" />

- **Branch (Ветка):** Coolify автоматически определит ветку в вашем репозитории.
- **Base Directory (Базовая директория):** введите директорию, которую Coolify должна использовать как корень. Используйте `/`, если ваши файлы находятся в корне, или укажите путь к подпапке (например, `/backend` для монорепозитория).

Нажмите кнопку **Continue**, когда все данные будут заполнены верно.

### 6. Настройте сетевые параметры

После нажатия **Continue** обновите такие настройки, как ваш домен и переменные окружения (при необходимости).

Важным параметром является порт, на котором работает ваше приложение.
Coolify устанавливает порт по умолчанию `3000`. Если ваше приложение слушает другой порт, обновите номер порта в сетевом разделе (network).

<ZoomableImage src="/docs/images/builds/packs/dockerfile/6.webp" alt="скриншот дашборда Coolify" />

## Продвинутые настройки

### Переменные окружения

Вы можете управлять переменными окружения через интерфейс Coolify.

Перейдите на вкладку **Environment Variables**, чтобы добавить или обновить их.

<ZoomableImage src="/docs/images/builds/packs/dockerfile/7.webp" alt="скриншот дашборда Coolify" />

### Команды до/после развертывания

<ZoomableImage src="/docs/images/builds/packs/dockerfile/8.webp" alt="скриншот дашборда Coolify" />

- **Pre-deployment (До развертывания):** при необходимости укажите скрипт или команду для выполнения в существующем контейнере перед началом деплоя. Команда запускается через `sh -c`, так что добавлять это вручную не нужно.
- **Post-deployment (После развертывания):** при необходимости укажите скрипт или команду для выполнения в новом собранном контейнере после завершения деплоя. Команда также выполняется через `sh -c`.

### Аргументы сборки (Build Arguments)

Coolify автоматически внедряет аргументы сборки в ваш Dockerfile во время процесса сборки. К ним относятся настроенные вами переменные окружения и предопределенные системные значения, такие как `SOURCE_COMMIT`.

Вы можете настроить эти параметры в меню **Advanced** вашего приложения.

#### Внедрение Build Args в Dockerfile

По умолчанию Coolify внедряет аргументы Docker-сборки (инструкции `ARG`) в ваш Dockerfile. Если вы предпочитаете управлять аргументами сборки вручную в самом Dockerfile, вы можете отключить это поведение в меню Advanced.

- **Enabled (по умолчанию):** Coolify автоматически внедряет аргументы сборки.
- **Disabled:** вы сами управляете инструкциями `ARG` в Dockerfile.

#### Включение хеша коммита (Source Commit) в сборку

Переменная `SOURCE_COMMIT` содержит хеш Git-коммита вашего исходного кода. По умолчанию она исключена из сборки для сохранения кэша сборки Docker. Вы можете включить её в меню Advanced, если это необходимо.

- **Disabled (по умолчанию):** `SOURCE_COMMIT` не включается, что улучшает использование кэша.
- **Enabled:** `SOURCE_COMMIT` включается как аргумент сборки.

::: warning Оптимизация кэша сборки
Включение «Include Source Commit in Build» приведет к тому, что кэш сборки Docker будет сбрасываться при каждом коммите, так как хеш коммита меняется каждый раз. Включайте это только в том случае, если процесс сборки требует наличия хеша коммита.
:::

## Известные проблемы и решения

::: details 1. При переходе по домену приложения отображается «No Available Server»
Если вы видите ошибку «No Available Server» при посещении сайта, это скорее всего связано с проверкой состояния (health check) вашего контейнера.

Запустите `docker ps` в терминале сервера, чтобы проверить, не является ли статус контейнера «unhealthy» или он всё еще запускается.

Чтобы решить эту проблему, исправьте причину, по которой контейнер не проходит проверку состояния, или отключите проверки состояния (health checks).
:::

::: details 2. Приложение работает только внутри контейнера
Если ваше приложение работает при проверке через `curl localhost` внутри контейнера, но вы получаете 404 или ошибку «No Available Server» при доступе по домену, проверьте настройки портов.

Убедитесь, что порт в сетевых настройках совпадает с портом, который слушает ваше приложение. Также проверьте логи запуска, чтобы убедиться, что приложение слушает не только localhost.

<ZoomableImage src="/docs/images/builds/packs/dockerfile/9.webp" alt="скриншот дашборда Coolify" />

При необходимости измените настройки приложения, чтобы оно слушало на всех интерфейсах (например, `0.0.0.0`).
:::
