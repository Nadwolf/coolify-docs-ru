# Приложения

Приложением может быть любой тип веб-приложения: статический сайт, приложение на NodeJS, PHP и так далее.

Для сложных приложений можно использовать развертывание на основе Docker Compose или готовые сервисы «в один клик».

## Как работает развертывание

Coolify развертывает все приложения в виде Docker-контейнеров. Это означает, что ваше приложение запускается в изолированном контейнере на вашем сервере.

**Ключевые концепции:**
- **Docker Image (Образ):** упакованная версия вашего приложения со всеми включенными зависимостями.
- **Container (Контейнер):** запущенный экземпляр вашего Docker-образа.
- **Build Process (Процесс сборки):** преобразует ваш исходный код в Docker-образ, готовый к развертыванию.

У вас есть два варианта развертывания приложений:

1. **Сборка в Coolify:** используйте [билдпаки (build packs)](/applications/build-packs/overview) для автоматического создания Docker-образов из вашего исходного кода.
2. **Использование готовых образов:** развертывайте существующие образы из реестров, таких как [Docker Hub](https://hub.docker.com/?utm_source=coolify.io) или [GitHub Container Registry](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry?utm_source=coolify.io).

::: tip Управление ресурсами
Сборка Docker-образов может быть ресурсоемкой. Вы можете использовать выделенный [сервер для сборки (build server)](/knowledge-base/server/build-server), чтобы вынести процесс сборки за пределы вашего продакшн-сервера.
:::

## Общая конфигурация

### Команды (Commands)

Вы можете переопределить команды по умолчанию, установив пользовательские значения в интерфейсе:

- Build (Сборка)
- Install (Установка)
- Start (Запуск)

::: info
  Если оставить эти поля пустыми, Nixpacks сам определит, какие команды нужно запустить. Например, для Nodejs он проверит lock-файлы и запустит `npm ci`, `yarn install` или `pnpm install` соответственно.
:::

### Базовая директория (Base Directory)

Полезно для монорепозиториев. Вы можете установить базовую директорию для всех команд, которые будут выполняться Coolify.

### Публичная директория (Public Directory)

Если вы собираете статический сайт, важно указать публичную директорию, чтобы сборщик знал, какую папку нужно обслуживать.

### Проброс портов (Port Exposes)

Это необходимо для того, чтобы Docker Engine знал, какие порты нужно открыть. Первый порт будет портом по умолчанию для проверки состояния (health checks).

Примеры:

Если у вас приложение на NodeJS, которое слушает порт 3000, установите: `3000`.
Если у вас приложение PHP-FPM, которое слушает порт 9000, установите: `9000`.
Если у вас сервер Nginx, который слушает порт 80, установите: `80`.

### Маппинг портов (Port Mappings)

::: warning
  При маппинге порта на хост-систему вы потеряете часть функционала, например, «Скользящие обновления» (Rolling Updates).
:::

Если вы хотите пробросить порт на хост-систему (сервер), вы можете сделать это так: `8080:80`.

Это сопоставит порт 8080 на сервере с портом 80 внутри контейнера.

::: info
  Если вам нужен прирост производительности и вам не нужен домен (например, сервер веб-сокетов с ОЧЕНЬ высоким трафиком), вы можете пробросить его порт на хост, чтобы запрос шел в обход прокси.
:::

## Продвинутые настройки

### Статический сайт (Static Site)

> Эта функция доступна только для билдпаков Nixpacks.

Если вам нужно обслуживать статический сайт (SPA, HTML и т. д.), вы можете установить значение `true`. Сайт будет обслуживаться через Nginx. `Отключено по умолчанию`.

### Принудительный HTTPS (Force HTTPS)

Если вы хотите принудительно использовать HTTPS (запретить HTTP-соединения), установите значение `true`. `Включено по умолчанию`.

### Автодеплой (Auto Deploy)

> Эта функция доступна только для репозиториев на базе GitHub App.

Если вы хотите автоматически развертывать приложение при пуше коммита в репозиторий, установите значение `true`. `Включено по умолчанию`.

### Предварительные развертывания (Preview Deployments)

Preview-деплои — отличный способ протестировать приложение перед слиянием с основной веткой. Представьте это как стейджинг-окружение.

#### Шаблон URL

Вы можете настроить URL для превью с помощью пользовательского шаблона. По умолчанию используется <span v-pre>`{{pr_id}}.{{domain}}`</span>.

Это означает, что если вы откроете Pull Request с ID `123`, а доменом ресурса является `example.com`, URL превью будет `123.example.com`.

:::success СОВЕТ
  Если у ресурса несколько доменов, первый из них будет использован в качестве части <span v-pre>`{{ domain }}`</span>.
:::

#### Автоматические Preview-деплои

> Эта функция доступна только для репозиториев на базе GitHub App.

Если вы хотите автоматически развертывать превью-версию приложения на основе Pull Request, установите значение `true`. `Отключено по умолчанию`.

Если установлено `true`, все PR, открытые в настроенную ветку ресурса, будут развернуты по уникальному URL.

#### Ручной запуск Preview-деплоев

Вы можете вручную развернуть Pull Request по уникальному URL, нажав кнопку `Deploy` на странице Pull Request.

### Git Submodules

Если вы используете гит-субмодули, установите значение `true`. `Включено по умолчанию`.

### Git LFS

Если вы используете Git LFS, установите значение `true`. `Включено по умолчанию`.

### Проверки состояния (Health Checks)

По умолчанию все контейнеры проверяются на жизнеспособность (liveness).

:::warning
  Прокси Traefik не будет работать, если для контейнера определен health check, но его статус — `unhealthy` (не здоров). Если вы не знаете, как настраивать проверки состояния, выключите их.
:::

### Откаты (Rollbacks)

Вы можете откатиться к предыдущей версии вашего ресурса. На данный момент поддерживаются только локальные образы, поэтому вы можете откатиться только к тому Docker-образу, который доступен локально.

### Лимиты ресурсов (Resource Limits)

По умолчанию контейнер не имеет ограничений по ресурсам. Вы можете установить лимиты здесь. Подробности читайте в [документации Docker](https://docs.docker.com/reference/compose-file/services).

## Типы развертывания

Доступно несколько типов развертывания приложений:

- Публичный Git-репозиторий
- Приватный Git-репозиторий ([GitHub App](https://docs.github.com/en/apps/using-github-apps/about-using-github-apps))
- Приватный Git-репозиторий ([Deploy Key](https://docs.github.com/en/authentication/connecting-to-github-with-ssh/managing-deploy-keys))
- На основе Dockerfile
- На основе Docker Compose
- На основе Docker Image

## Билдпаки (Build Packs)

Билдпаки помогают преобразовывать ваш исходный код в Docker-образы. Coolify поддерживает несколько вариантов билдпаков для разных нужд:

- **[Nixpacks](/applications/build-packs/nixpacks)** — автоматическое обнаружение и сборка (рекомендуется для большинства приложений).
- **[Static](/applications/build-packs/static)** — для статических сайтов и SPA.
- **[Dockerfile](/applications/build-packs/dockerfile)** — использование собственного кастомного Dockerfile.
- **[Docker Compose](/applications/build-packs/docker-compose)** — для мультисервисных приложений.
- **Docker Image** — развертывание готовых образов из реестров.

Для подробного ознакомления с каждым билдпаком см. раздел [Билдпаки](/applications/build-packs/overview).

:::tip Быстрый старт
Coolify использует [Nixpacks](https://nixpacks.com) по умолчанию. Он автоматически определяет тип вашего приложения и собирает его соответствующим образом. Для большинства приложений вам не потребуется ничего настраивать.
:::
