# Ошибки Gateway Timeout (504)

Ошибки тайм-аута шлюза (Gateway Timeout) возникают, когда прокси-сервер Coolify не может получить ответ от вашего приложения в течение установленного периода ожидания. Это отличается от ошибок [Bad Gateway (502)](/troubleshoot/applications/bad-gateway#bad-gateway-502-error), которые означают, что прокси вообще не может подключиться к вашему приложению.

## Основные причины

В Coolify существует два основных сценария, вызывающих ошибки 504 Gateway Timeout:

1. **Изоляция пользовательской сети Docker** — прокси не может достучаться до приложений, использующих кастомные сети.
2. **Тайм-ауты при загрузке/скачивании больших файлов** — стандартные настройки тайм-аута слишком коротки для передачи больших объемов данных.

## Проблема 1: Изоляция пользовательской сети Docker

### Симптомы

- Приложение работает сразу после развертывания.
- Ошибки 504 Gateway Timeout появляются через несколько часов или дней.
- Приложение доступно напрямую по IP и порту (требуется ручной маппинг порта).
- Перезапуск приложения временно решает проблему.
- В конфигурации используются пользовательские (custom) сети Docker.

### Первопричина

Когда вы определяете пользовательские сети в файле Docker Compose, контейнер `coolify-proxy` работает в собственных сетях Coolify, в то время как ваше приложение — в кастомной сети. Эта изоляция мешает прокси связаться с приложением, особенно когда внутренний DNS Docker возвращает разные IP-адреса в зависимости от времени и моментов подключения к сетям.

### Диагностика

1. **Проверьте, использует ли ваше приложение кастомные сети:**

   ```bash
   docker inspect <имя-вашего-контейнера> --format='{{range $k,$v := .NetworkSettings.Networks}}Network: {{$k}}, IP: {{$v.IPAddress}}, Gateway: {{$v.Gateway}}{{println}}{{end}}'
   ```

2. **Проверьте сетевые подключения прокси:**

   ```bash
   docker inspect coolify-proxy --format='{{range $k,$v := .NetworkSettings.Networks}}Network: {{$k}}, IP: {{$v.IPAddress}}, Gateway: {{$v.Gateway}}{{println}}{{end}}'
   ```

### Решения

#### Решение 1: Используйте "Destinations" в Coolify (рекомендуется)

Позвольте Coolify автоматически управлять сетями, используя [Destinations](/knowledge-base/destinations/) вместо кастомных сетей:

1. Удалите определения кастомных сетей из вашего файла Docker Compose.
2. [Настройте сетевое назначение (destination)](/knowledge-base/destinations/create) в интерфейсе Coolify в разделе **Destinations**.
3. [Перенесите ваше приложение/сервис в нужное назначение](/knowledge-base/destinations/manage#assign-resources-to-a-destination).
4. Выполните повторное развертывание.

**До (проблемный вариант):**

```yaml
services:
  app:
    image: myapp:latest
    networks:
      - custom-network

networks:
  custom-network:
    driver: bridge
```

**После (рекомендуемый вариант):**

```yaml
services:
  app:
    image: myapp:latest
    # Пусть Coolify управляет сетью
```

#### Решение 2: Ручное подключение сети (временное)

Если вам необходимо использовать кастомные сети, вручную подключите прокси:

```bash
docker network connect <имя-вашей-сети> coolify-proxy
```

**Примечание:** Это временное исправление, которое может потребоваться повторить после перезапуска прокси.

## Проблема 2: Тайм-ауты при загрузке/скачивании больших файлов

### Симптомы

- Ошибки 504 при загрузке больших файлов (>100 МБ).
- Ошибки 504 при отправке больших Docker-образов в реестр.
- Ошибки 504 во время длительных запросов (>60 секунд для Traefik/Nginx).
- Маленькие файлы и быстрые запросы работают нормально.

### Первопричина

Стандартное поведение тайм-аутов зависит от вашего прокси:

- **Traefik**: Тайм-аут чтения по умолчанию составляет 60 секунд.
- **Caddy**: Тайм-аутов по умолчанию нет (запросы могут выполняться бесконечно).
- **Nginx** (базы данных «в один клик»): Тайм-аут по умолчанию составляет 60 секунд.

Любой запрос, превышающий настроенный тайм-аут, приведет к ошибке 504 Gateway Timeout, даже если приложение на бэкенде всё еще обрабатывает его.

### Диагностика

1. **Проверьте, какой прокси вы используете:**

   - Перейдите в **Servers > [ВашСервер] > Proxy** в интерфейсе Coolify.
   - Тип прокси (Traefik, Caddy и т. д.) будет отображен там.

2. **Проверьте текущую конфигурацию прокси:**

   - В том же разделе **Proxy** поищите настройки тайм-аутов.
   - Проверьте настройки приложений/сервисов на наличие кастомных меток (labels), которые могут переопределять стандартные значения.

3. **Следите за длительностью запросов в логах приложения:**

   - Перейдите в логи вашего приложения/сервиса в Coolify.
   - Ищите длительные запросы, превышающие 60 секунд.

4. **Протестируйте с файлом меньшего размера, чтобы подтвердить зависимость от размера.**

### Решения

#### Решение 1: Увеличьте тайм-аут прокси

Способ настройки зависит от типа прокси:

##### Для Traefik (прокси по умолчанию)

Добавьте пользовательскую конфигурацию Traefik для увеличения тайм-аута. У вас есть несколько вариантов в зависимости от потребностей:

Перейдите в настройки прокси вашего сервера и добавьте новые тайм-ауты в разделе команд (command):

```yaml
command:
  - '--entrypoints.https.transport.respondingTimeouts.readTimeout=5m'
  - '--entrypoints.https.transport.respondingTimeouts.writeTimeout=5m'
  - '--entrypoints.https.transport.respondingTimeouts.idleTimeout=5m'
```

Подробнее о тайм-аутах Traefik читайте в [официальной документации](https://doc.traefik.io/traefik/reference/install-configuration/entrypoints/#timeout).

##### Для Caddy

Так как у Caddy нет тайм-аута по умолчанию, вы обычно не столкнетесь с этой проблемой. Однако, если вам нужно УСТАНОВИТЬ тайм-аут (для безопасности или управления ресурсами):

1. Добавьте следующее в **Container Labels** в Coolify:

```yaml
# Установить 5-минутный тайм-аут (300 секунд)
caddy.servers.timeouts.read_body=300s
caddy.servers.timeouts.read_header=300s
caddy.servers.timeouts.write=300s
caddy.servers.timeouts.idle=5m
```

Подробнее о тайм-аутах Caddy читайте в [официальной документации](https://caddyserver.com/docs/caddyfile/options#timeouts).

##### Для Nginx (базы данных «в один клик»)

Конфигурацию Nginx для баз данных в Coolify нельзя изменить напрямую. Вместо этого **обойдите Nginx**, чтобы избежать проблем с тайм-аутами:

1. Перейдите в настройки вашей базы данных в Coolify.
2. **Выключите** опцию "Make it publicly available?" (Сделать доступной публично).
3. Используйте **Port Mappings**, чтобы пробросить порт базы данных напрямую, и перезапустите базу.
4. Это пробросит порт прямо из контейнера, минуя Nginx и его ограничения по времени ожидания.

**Пример:** Для базы PostgreSQL пробросьте порт `5432:5432` для прямого доступа без тайм-аутов прокси.

Подробнее о публичном доступе к БД: [Базы данных в один клик](/databases/#ports-mapping-vs-public-port).

#### Решение 2: Используйте поочередную загрузку (Chunked Uploads)

Для очень больших файлов рассмотрите возможность реализации поочередной загрузки (chunked uploads) в вашем приложении:

1. Разделите большие файлы на мелкие части на стороне клиента.
2. Загружайте части по отдельности (каждая должна укладываться в лимит тайм-аута).
3. Соберite их на стороне сервера.

#### Решение 3: Используйте фоновую обработку

Для длительных операций:

1. Примите запрос и сразу верните ID задачи.
2. Обработайте запрос в фоновом режиме.
3. Предоставьте эндпоинт для проверки статуса задачи.

## Краткий чеклист диагностики

Пройдите по этим шагам, чтобы выявить конкретную проблему:

1. **Проверьте код ошибки:**
   - 504 = Gateway Timeout (это руководство).
   - 502 = Bad Gateway (см. [устранение неполадок Bad Gateway](/troubleshoot/applications/bad-gateway)).

2. **Проверьте время:**
   - Сразу = скорее всего, проблема в сети или конфигурации.
   - Примерно через 60 секунд = скорее всего, проблема в тайм-ауте.
   - Случайно через часы/дни = скорее всего, проблема в изоляции сети.

3. **Проверьте конфигурацию сети:**

   ```bash
   # Список всех сетей Docker
   docker network ls

   # Проверьте, какие сети используют ваши контейнеры
   docker ps --format "table {{.Names}}\t{{.Networks}}"
   ```

## Советы по предотвращению

1. **Избегайте кастомных сетей Docker**, если это не является жизненно необходимым.
2. **Устанавливайте подходящие тайм-ауты** в соответствии с нуждами вашего приложения на этапе настройки.
3. **Настраивайте проверки состояния (health checks)** для поддержания связи.
4. **Регулярно проверяйте логи прокси** на предмет паттернов тайм-аутов.
5. **Используйте индикаторы прогресса** для длительных операций, чтобы предотвратить тайм-ауты на стороне клиента.

## Поддержка

Если эти решения не помогли:

1. Соберите диагностическую информацию:

   ```bash
   # Сохраните этот вывод
   docker ps --format "table {{.Names}}\t{{.Networks}}\t{{.Status}}"
   docker logs coolify-proxy --tail 200 > proxy-logs.txt
   docker logs <имя-вашего-контейнера> --tail 200 > app-logs.txt
   ```

2. Присоединяйтесь к нашему [Discord-сообществу](https://coolify.io/discord).
3. Поделитесь конфигурацией, логами и списком предпринятых шагов.
