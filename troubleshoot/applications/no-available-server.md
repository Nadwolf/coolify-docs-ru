# Ошибка No Available Server (503)

Если ваше развернутое приложение или сервис показывает ошибку **"No Available Server"**, это означает, что [Traefik](/knowledge-base/proxy/traefik/overview) (обратный прокси) не может найти ни одного (исправного) контейнера для маршрутизации трафика по указанному защищенному URL (`https`).

## Что вызывает эту ошибку?

Ошибка "No Available Server" возникает, когда:

1. **Неудачные проверки состояния (Health Checks)** — Traefik считает ваш контейнер «нездоровым» (unhealthy).
2. **Проблемы с конфигурацией домена** — неправильная настройка домена или отсутствие вариантов с www / без www.
3. **Несоответствие портов** — указанные порты не совпадают с фактическим портом, который слушает ваше приложение.
4. **Простой при развертывании** — кратковременный простой во время обновления контейнеров.
5. **Проблемы самого Traefik** — например, несоответствие версий Docker API.

## Быстрая диагностика

### 1. Проверьте статус здоровья контейнера

Сначала убедитесь, что ваши контейнеры запущены и исправны:

```bash
# Подключитесь по SSH к серверу и проверьте статус
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

Ищите контейнеры со статусом `(unhealthy)` — это указывает на проблему с проверкой состояния.

### 2. Проверьте конфигурацию домена

Убедитесь, что домен правильно указан в настройках приложения/сервиса и что ваши DNS-записи указывают на правильный IP-адрес. Ознакомьтесь с документацией по [доменам](/knowledge-base/domains) для понимания правил форматирования.

### 3. Проверьте логи прокси Traefik

Изучите логи Traefik на наличие внутренних ошибок:

Через интерфейс Coolify:
- Перейдите в `Servers` → `[Ваш сервер]` → `Proxy` → `Logs`

Или через SSH:
```bash
# Просмотр логов прокси (последние 50 строк)
docker logs coolify-proxy --tail 50
```

Ищите сообщения об ошибках, например `client version 1.24 is too old`, что указывает на [несоответствие версии Docker API](#исправление-traefik-из-за-ошибки-версии-docker-api).

## Типичные решения

### Исправление неудачных проверок состояния (самое частое)

**Симптомы:**
- Контейнер отображается как `(unhealthy)` в Docker.
- Путь проверки состояния возвращает ошибку.
- В контейнере отсутствуют необходимые зависимости (например, `curl` или `wget`).

**Шаги:**

1. **Временно отключите проверку состояния:**
   - Перейдите в конфигурацию приложения.
   - Отключите Health Check.
   - Перезапустите приложение.
   Если это решило проблему, значит, дело именно в настройках проверки состояния.

2. **Исправьте настройки Health Check:**

   ::: info
   Ниже приведены распространенные способы исправить проверку состояния. Скорректируйте их под ваше приложение. Подробнее: [Настройка Health Checks](/knowledge-base/health-checks#configure-health-checks).
   :::

   **Отсутствие зависимостей:**
   Убедитесь, что в вашем Docker-образе установлены инструменты, необходимые для проверки. Для [приложений](/applications/index) обычно нужны `curl` или `wget`.

   ```dockerfile
   # Добавьте curl в ваш Dockerfile
   RUN apt-get update && apt-get install -y curl
   # ИЛИ для образов на базе Alpine
   RUN apk add --no-cache curl
   ```

   **Неверный путь или хост:**
   Убедитесь, что приложение действительно обслуживает эндпоинт проверки и не возвращает ошибку. В большинстве случаев хост должен быть `localhost` или `127.0.0.1`.

   **Несоответствие портов:**
   - Порт проверки должен совпадать с портом, который слушает приложение.
   - Если приложение работает на порту `3000`, проверка состояния также должна идти на порт `3000`.

3. **Проверьте Health Check вручную:**
   Если вышеперечисленное не помогло, выполните команду проверки вручную внутри контейнера (вкладка Terminal в Coolify или через SSH):

   ```bash
   # Пример проверки через curl внутри контейнера
   docker exec -it <имя-контейнера> curl -f http://localhost:3000/health
   ```

### Исправление конфигурации домена

#### Неправильная настройка домена
**Симптомы:** Работает с автогенерируемым доменом (например, `sslip.io`), но не с вашим собственным.
**Шаги:** Убедитесь, что домен правильно настроен в Coolify и у вашего DNS-провайдера согласно руководству по [доменам](/knowledge-base/domains).

#### Проблемы с редиректом
**Симптомы:** Настроен редирект (на www или без www), работает для одного варианта, но не для другого.
**Шаги:**
1. **Добавьте оба варианта домена:** Укажите и основной домен, и вариант с www в поле `Domains`: `https://example.com,https://www.example.com`
2. **Настройте редирект:** Установите `Direction` в значение `Allow www & non-www`, если хотите, чтобы работали оба варианта.
3. **Перезапустите приложение.**

#### Проблемы с HTTPS
Если сайт доступен по HTTP, но не по HTTPS:
- **Для HTTPS с SSL:** Используйте префикс `https://`: `https://example.com`.
- **Только для HTTP:** Используйте префикс `http://`: `http://example.com` (SSL-сертификат не будет сгенерирован).
После изменений перезапустите приложение.

### Исправление конфигурации портов

**Симптомы:**
- Приложение работает через `http://IP:порт`, но не через домен.
- [Traefik](/knowledge-base/proxy/traefik/overview) не может достучаться до приложения.

**Шаги:**

1. **Проверьте проброшенный порт (Exposed Port):**
   Прокси должен знать, какой порт слушает ваше приложение.
   <ZoomableImage src="/docs/images/troubleshoot/applications/bad-gateway/1.webp" alt="Скриншот No Available Server" />
   В [приложениях](/applications/index) это задается в поле `Port Exposes`.
   <ZoomableImage src="/docs/images/troubleshoot/applications/bad-gateway/4.webp" alt="Скриншот No Available Server" />
   В **сервисах** это задается либо добавлением порта в конце URL (например, `https://example.com:3000`), либо директивой `EXPOSE` в `Dockerfile`.

2. **Проверьте адрес прослушивания в приложении:**
   <ZoomableImage src="/docs/images/troubleshoot/applications/bad-gateway/3.webp" alt="Скриншот No Available Server" />
   Убедитесь, что ваше приложение не привязано только к `localhost` или `127.0.0.1`. Оно должно слушать на всех интерфейсах (`0.0.0.0`).

### Обработка простоя при развертывании

**Симптомы:** Кратковременная ошибка "No Available Server" только во время обновления контейнеров.
**Решение:** Настройте плавное обновление (Rolling Updates). См. [подробнее здесь](/knowledge-base/rolling-updates).

### Исправление Traefik из-за ошибки версии Docker API

**Симптомы:**
- Ошибка "No Available Server".
- В логах прокси Coolify видна ошибка: `Error response from daemon: client version 1.24 is too old. Minimum supported API version is 1.44...`
- Контейнер исправен, но домен не открывается.

**Причина:** Ранее в Traefik версии API были жестко прописаны.
**Решение:**
Начиная с версий **v2.11.31** и **v3.6.1**, Traefik умеет автоматически договариваться о версии API. Вам нужно обновить Traefik вручную:
1. Перейдите в дашборд Coolify → Servers → [Ваш сервер] → Proxy → Configuration.
2. Измените версию Traefik на `v3.6.1` (или `v2.11.31`, если используете v2).
3. Нажмите `Restart Proxy`.
<ZoomableImage src="/docs/images/troubleshoot/applications/bad-gateway/no-available-server/update-traefik-version.webp" alt="Обновление версии Traefik в Coolify" />

*Примечание:* Это нужно сделать на каждом сервере, подключенном к вашему экземпляру Coolify.

## Расширенная отладка

### Проверка конфигурации Traefik
```bash
# Просмотр динамической конфигурации Traefik
cat /data/coolify/proxy/dynamic/*.yml

# Просмотр меток контейнера для проверки маршрутизации
docker inspect <имя-контейнера> --format='{{json .Config.Labels}}' | jq
```

### Проверка логов приложения
```bash
docker logs <имя-контейнера> -f
```

### Тестирование изнутри контейнера
```bash
docker exec -it <имя-контейнера> /bin/sh
curl -f http://localhost:3000/health
```

## Советы по предотвращению
1. **Всегда используйте проверки состояния (Health Checks):** реализуйте эндпоинт `/health` и убедитесь в наличии `curl`/`wget` в образе.
2. **Сначала тестируйте локально:** проверьте эндпоинт и порты перед деплоем.
3. **Следите за статусом контейнеров:** регулярно проверяйте `docker ps`.
4. **Используйте Staging-окружение** для проверки конфигураций доменов.
