# Постоянное хранилище (Persistent Storage)
Вы можете добавить постоянное хранилище к своим ресурсам, чтобы безопасно сохранять данные (файлы, базы данных) между развертываниями и обновлениями контейнеров.

Тип хранилища может отличаться в зависимости от места назначения (Destination), куда вы развертываете приложение.

## Docker Engine

При использовании Docker Engine в качестве среды выполнения, постоянное хранилище может быть представлено в двух формах: `volume` (том Docker, управляемый самим Docker) или `bind mount` (прямое монтирование файла или директории с хост-системы — то есть с файловой системы вашего сервера).

### Volume (Том)

Для создания тома Docker вам нужно определить:

- `Name` — желаемое имя тома.
- `Destination Path` — абсолютный путь внутри контейнера, куда будет примонтирован (подключен) том.

::: warning Осторожно
  Базовая директория внутри контейнера по умолчанию обычно `/app`. Если вам, например, нужно хранить загруженные файлы в папке `storage`, укажите `/app/storage` в качестве пути назначения (`Destination Path`).
:::

::: success Совет
  Чтобы избежать случайного пересечения данных и конфликтов между разными ресурсами, Coolify автоматически добавляет уникальный идентификатор (UUID) ресурса в начало имени тома.
:::

### Bind Mount (Монтирование директории с хоста)

Для создания bind mount (монтирования) вам нужно определить:

- `Name` — имя тома, которое будет использоваться как внутренняя ссылка или идентификатор настройки.
- `Source Path` — абсолютный путь на хост-системе (непосредственно на самом сервере Linux). **В этом случае физический том Docker (Volume) не создается.**
- `Destination Path` — абсолютный путь внутри контейнера, куда будет примонтирована указанная директория.

::: warning Осторожно
  Точно так же, базовая директория внутри контейнера обычно `/app`. Если вам нужно прокинуть файлы в папку `storage`, укажите `/app/storage` в качестве пути назначения.
:::

::: warning Важное предупреждение!
  Общий файл/директория для нескольких контейнеров одновременно? **ЭТО КРАЙНЕ НЕ РЕКОМЕНДУЕТСЯ.**

  Если вы монтируете один и тот же файл или папку ("bind mount") в несколько одновременно запущенных контейнеров с правом на запись, вы должны убедиться на 100%, что в коде ваших приложений реализован корректный и надежный механизм блокировки файлов (file locking). В противном случае неизбежно повреждение данных файлов.
:::
