# Резервное копирование и восстановление вашего экземпляра Coolify

Это руководство покажет вам, как создать резервную копию вашего экземпляра Coolify и восстановить его на новом сервере.

Существует два основных метода создания бэкапов:

- **S3 Бэкап:** использование S3-совместимого хранилища для автоматического сохранения резервных копий.
- **Ручной бэкап:** запуск процесса создания резервной копии вручную из панели управления Coolify.

Если вы используете S3-совместимое хранилище, просто скачайте файл бэкапа у вашего S3-провайдера и перенесите его на новый сервер.

Остальная часть этого руководства посвящена методу ручного бэкапа, который подходит большинству пользователей.

::: warning ВНИМАНИЕ:
Это руководство касается только резервного копирования и восстановления самого экземпляра Coolify — но не данных ваших приложений.

Все настройки из панели управления Coolify будут восстановлены, но данные приложений (например, тома/mounts) необходимо сохранять и восстанавливать вручную.

Для подробной информации обратитесь к [Руководству по миграции приложений](https://coolify.io/docs/knowledge-base/how-to/migrate-apps-different-host).
:::

## 1. Создание ручного бэкапа

1. **Перейдите на страницу бэкапа в дашборде:**  
   В панели Coolify нажмите на **Settings** и выберите вкладку **Backup**, чтобы просмотреть настройки базы данных и резервного копирования.
   <ZoomableImage src="/docs/images/knowledge-base/how-tos/backup-restore-coolify/1.webp" alt="Скриншот резервного копирования Coolify" />

2. **Запустите бэкап:**  
   Нажмите кнопку **Backup Now**. Это запустит процесс создания резервной копии в фоновом режиме.
   <ZoomableImage src="/docs/images/knowledge-base/how-tos/backup-restore-coolify/2.webp" alt="Скриншот резервного копирования Coolify" />

3. **Скачайте или скопируйте путь к бэкапу:**  
   Как только процесс завершится, в интерфейсе появится кнопка **Download** и путь к файлу.
   <ZoomableImage src="/docs/images/knowledge-base/how-tos/backup-restore-coolify/3.webp" alt="Скриншот резервного копирования Coolify" />
   - **Download:** сохраняет файл бэкапа на ваш локальный компьютер.
   - **Copy Path:** вы можете использовать этот путь, например, с инструментом SCP, чтобы перенести файл напрямую на новый сервер.

::: info Примечание
Если для бэкапов используется S3-совместимое хранилище, скачайте файл резервной копии от вашего S3-провайдера.
:::

## 2. Получение вашего `APP_KEY`

Перед восстановлением бэкапа вам необходимо получить `APP_KEY` вашего текущего инстанса Coolify. Этот ключ используется для расшифровки ваших данных во время восстановления.

1. **Откройте терминал в Coolify:**  
   Перейдите во вкладку **Terminal** в дашборде и подключитесь к серверу, на котором запущен Coolify (по умолчанию он называется `localhost`).

2. **Просмотрите файл окружения (.env):**  
   Выполните следующую команду, чтобы вывести содержимое файла `.env`:
   ```sh
   cat /data/coolify/source/.env
   ```
   Скопируйте значение `APP_KEY` и сохраните его в надежном месте. Этот ключ критически важен для процесса восстановления.

::: danger ВАЖНО
Надежно сохраните `APP_KEY`. Без него вы не сможете восстановить свои данные из бэкапа.
:::

## 3. Резервное копирование SSH-ключей Coolify

Coolify генерирует один (или несколько) файлов SSH-ключей в директории `/data/coolify/ssh/keys`. Если вы восстанавливаете Coolify на новой машине, вы должны перенести эти файлы ключей, иначе вы не сможете подключиться к управляемым серверам.

1. **Найдите SSH-ключи на старом хосте:**

   ```sh
   ls -l /data/coolify/ssh/keys
   ```

   Вы увидите один или несколько файлов с именами вида:

   ```sh
   ssh_key@<random_id1>
   ssh_key@<random_id2>
   ssh_key@<random_id3>
   ```

   Каждая запись `ssh_key@…` представляет собой ключ ED25519, который Coolify использует для доступа к вашим серверам по SSH.

2. **Скопируйте все эти файлы ключей на ваш новый сервер и сохраните их в безопасном месте.**

3. **Перенос публичных ключей:**

   Возьмите публичные ключи из файла `~/.ssh/authorized_keys` вашего старого сервера и добавьте их в аналогичный файл `~/.ssh/authorized_keys` на новом сервере (не заменяйте существующий контент, просто добавьте в конец).

## 4. Подготовка нового сервера

Подготовьте новый сервер, на котором будет развернут восстановленный экземпляр Coolify.

1. **Установите чистый экземпляр Coolify:**  
   Следуйте [инструкциям по установке](/get-started/installation).

   Обязательно укажите правильный номер версии (например, `-s 4.0.0-beta.400`) в конце скрипта установки, чтобы версия совпадала с предыдущей.

   Пример для версии `4.0.0-beta.400`:

   ```sh
   curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash -s 4.0.0-beta.400
   ```

   Замените `4.0.0-beta.400` на вашу версию.

2. **Проверьте установку:**  
   Откройте новый интерфейс Coolify в браузере. Вы должны увидеть страницу регистрации, что означает отсутствие данных.

## 5. Перенос и восстановление бэкапа

1. **Перенесите файл бэкапа:**  
   Скопируйте файл резервной копии и SSH-ключи на новый сервер (через SCP, FTP или любой другой безопасный способ).
2. **Остановите Coolify:**

   ```sh
   docker stop coolify coolify-redis coolify-realtime coolify-proxy
   ```

3. **Запустите команду восстановления:**  
   Используйте инструмент восстановления PostgreSQL, чтобы импортировать бэкап в контейнер базы данных.

   ```sh
   cat /путь/к/файлу_бэкапа \
     | docker exec -i coolify-db \
       pg_restore --verbose --clean --no-acl --no-owner -U coolify -d coolify
   ```

   Замените `/путь/к/файлу_бэкапа` на реальный путь к файлу на сервере.

   ::: warning Примечание:
   Могут появиться предупреждения о существующих внешних ключах или последовательностях (sequences), их обычно можно игнорировать, если базовая структура данных сохранена.
   :::

## 6. Замена автоматически сгенерированного SSH-ключа

Замените файлы ключей в директории `/data/coolify/ssh/keys`.

1. **Удалите любые автоматически созданные ключи:**

```sh
rm -f /data/coolify/ssh/keys/*
```

2. **Переместите ваши старые ключи в** `/data/coolify/ssh/keys/`

   Это те самые файлы, которые вы скопировали на новый сервер на [шаге 3](#3-резервное-копирование-ssh-ключей-coolify).

## 7. Обновление настроек окружения (Restoration)

После восстановления бэкапа обновите конфигурацию окружения, чтобы новый экземпляр смог использовать старые данные.

1. **Отредактируйте файл окружения:**  
   Откройте файл `.env`:

   ```sh
   nano /data/coolify/source/.env
   ```

2. **Добавьте предыдущий APP Key:**
   Добавьте новую переменную окружения `APP_PREVIOUS_KEYS` и вставьте туда значение `APP_KEY`, которое вы сохранили ранее. Если вы переезжали несколько раз, ключи можно перечислить через запятую (без пробелов):

   ```yaml
   # Пример для одного ключа:
   APP_PREVIOUS_KEYS=ваш_предыдущий_app_key_здесь
   ```

   ```yaml
   # Пример для нескольких ключей:
   APP_PREVIOUS_KEYS=app_key_1,app_key_2,app_key_3
   ```

   Сохраните изменения и выйдете из редактора.

## 8. Перезапуск Coolify

Чтобы применить восстановленный бэкап и новые настройки, перезапустите Coolify с помощью скрипта установки.

1. **Запустите скрипт установки:**  

   ```sh
   curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash -s 4.0.0-beta.400
   ```

   Замените `4.0.0-beta.400` на нужную версию.

2. **Проверьте результат:**  
   Зайдите в дашборд Coolify. Войдите с учетными данными вашего старого инстанса. Ваши проекты, деплои и настройки должны быть на месте.

## Поиск и устранение неисправностей

- **Ошибка 500 при входе или доступе к проекту:**  
  Перепроверьте, правильно ли задана переменная `APP_PREVIOUS_KEYS` в файле `.env`.

- **Ошибки прав доступа (Permission Denied):**  
  Если возникают проблемы с доступом к директориям, смените владельца папки `/data/coolify`. Так как Coolify использует пользователя root, убедитесь, что владельцем является **root**:
   ```sh
   sudo chown -R root:root /data/coolify
   ```
- **Сервер недоступен (Permission denied):**
  Если Coolify не может подключиться к серверам по SSH из-за отсутствия нужных файлов ключей.

  Убедитесь, что вы скопировали ВСЁ содержимое `/data/coolify/ssh/keys/` со старого хоста на новый. Если файлы не совпадают точно, возникнет ошибка.

  Также убедитесь, что соответствующий публичный ключ из файла `authorized_keys` старого хоста был добавлен в `~/.ssh/authorized_keys` нового хоста.
