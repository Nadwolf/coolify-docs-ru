# Скользящие обновления в Coolify (Rolling Updates)

Скользящие обновления (Zero Downtime Deployments) позволяют Coolify незаметно обновлять ваше приложение: система запускает новый контейнер с обновленным кодом и лишь затем плавно останавливает текущий, работающий контейнер.

Этот подход во много раз минимизирует время простоя (downtime) и гарантирует, что ваш сервис остается полностью доступным для пользователей в процессе развертывания новой версии.

## Как работают скользящие обновления

При запуске нового деплоя (обновления) Coolify запускает **новый экземпляр** контейнера, в то время как **старый (существующий)** контейнер продолжает работать и обслуживать запросы.

Как только новый контейнер успешно подтвердит свое "здоровье" (получит статус *healthy* благодаря проверке состояния), старый контейнер будет корректно остановлен, а трафик начнет перенаправляться на новую версию.
Этот процесс плавного замещения и называется *скользящим обновлением*, он помогает полностью избежать прерывания обслуживания.

## Условия для работы скользящих обновлений

Для корректной работы скользящих обновлений (Rolling Updates) обязательно должны быть выполнены все следующие условия:

- **Настройка проверки состояния (Health Check):** Для приложения должна быть настроена хотя бы одна валидная проверка состояния, которая успешно проходит в контейнере.
    - Проверка гарантирует, что новый контейнер полностью поднялся, запустил все процессы и готов принимать трафик до того, как старый контейнер будет убит.
    - Без корректного настроенного Health Check процесс деплоя не может подтвердить готовность контейнера, что может привести к сбоям маршрутизации (ошибкам 502/503).

- **Имя контейнера по умолчанию:** Скользящие обновления требуют обязательного использования стандартных (генерируемых Coolify) имен контейнеров.
    - Если вы жестко зададите кастомное имя контейнера, процесс обновления гарантированно завершится с ошибкой вида "Имя занято", что сделает скользящее обновление абсолютно невозможным.

- **Не используйте Docker Compose:** Скользящие обновления не поддерживаются для деплоев на основе чистого Docker Compose `docker-compose.yaml` (Compose Buildpack).
    - В Docker Compose по умолчанию используются статические предсказуемые имена контейнеров, что мешает правильному одновременному запуску двух экземпляров приложения при обновлении.

- **Проброс портов на хост (Port Mapping):** Не используйте жесткое связывание портов (проброс порта) контейнера с портом на сервере (хосте).
    - Новый контейнер физически не сможет занять тот же самый порт на сервере (например, `8080:8080`) во время процесса обновления, пока старый контейнер все еще его "держит".
    - Это вызовет конфликт портов Docker при попытке запуска, и скользящее обновление гарантированно завершится крахом.

## Настройка проверок состояния (Health Checks)

Чтобы обеспечить 100% успешные скользящие обновления, обязательно убедитесь, что в вашем приложении реализован специальный эндпоинт для проверки состояния (health status endpoint).

Этот эндпоинт должен возвращать успешный ответ (например, HTTP-статус `200 OK`), когда контейнер работает нормально и приложение загрузилось. Подробнее о настройке конфигурации читайте в нашем специализированном [руководстве по проверке состояния](/knowledge-base/health-checks).

## Устранение неполадок (Troubleshooting)

Если у вас возникли проблемы со скользящими обновлениями, выполните следующие шаги:

1. **Проверьте настройки Health Check:** Убедитесь, что эндпоинт проверки состояния реализован в коде правильно, конфигурация в Coolify указана верно, внутри контейнера стоит утилита `curl` или `wget` и проверка возвращает валидный ответ. Неудачная проверка (контейнер в статусе unhealthy) навсегда остановит процесс обновления старой версии до фикса бага.

2. **Проверьте именование контейнеров:** Убедитесь, что вы используете стандартную, заложенную Coolify конвенцию именования контейнеров. При ручном заданном кастомном имени скользящие обновления работать не будут.

3. **Изучите логи:** Внимательно проверьте логи системного контейнера `coolify-proxy`, логи процесса деплоя и логи самого приложения на наличие сообщений об ошибках, связанных с процессом запуска (booting / starting up). Это может дать точную подсказку, что именно мешает завершению операции обновления (например, падение нового контейнера из-за неверного синтаксиса или бага в коде).
