# Масштабируемость (Scalability)

Если вашему приложению требуется балансировка нагрузки или высокая доступность (high availability), в Coolify есть несколько вариантов:

- [Традиционное горизонтальное масштабирование](#традиционное-горизонтальное-масштабирование)
- [Docker Swarm](#docker-swarm)

::: info Совет профессионала
 Вам не нужно масштабировать приложение, если у вас 0 пользователей. Начинайте с малого и масштабируйтесь по мере роста пользовательской базы!
:::

## Традиционное горизонтальное масштабирование

При традиционном горизонтальном масштабировании вы развертываете свои приложения на нескольких серверах и используете балансировщик нагрузки для распределения трафика между ними.

Это самый распространенный тип масштабирования, он прост в понимании и реализации.

### Требования Coolify
1. **Добавление серверов**
     - Вам нужно добавить и валидировать серверы в Coolify.
2. **Настройка реестра контейнеров (Docker Registry)** для вашего приложения
     - **Зачем?** Поскольку нескольким серверам понадобится доступ к одному и тому же собранному образу, он должен храниться в общем месте.

### Требования к инфраструктуре
1. **Балансировщик нагрузки (Load balancer)**
2. **Брандмауэр (Firewall)** — опционально, но рекомендуется.

### Примеры

Какой вариант лучше?

Это зависит от ваших потребностей, но мы рекомендуем вариант `один домен на несколько серверов`.

::: info Совет
Мы также рекомендуем использовать [Hetzner](https://coolify.io/hetzner) (реферальная ссылка) для серверов.

**(Облачная версия Coolify и все наши другие сервисы также используют Hetzner).**
:::

#### Один домен на несколько серверов (Рекомендуется)

![load-balance-one-domain](https://coolify.io/docs/images/loadbalancer/one-domain.webp)

- **Плюсы**:
    - Легко понять и реализовать.
    - Легко управлять.
    - Легко масштабировать.
    - На серверах не требуется прокси.
    - Проверки состояния (healthchecks) доступны для конкретного приложения, а не для сервера в целом.
- **Минусы**:
    - Вам нужен брандмауэр на каждом сервере, чтобы предотвратить прямой доступ к ним (у большинства VPS-провайдеров есть программные брандмауэры).

#### Несколько доменов на несколько серверов (Рекомендуется, но сложнее)

![load-balance-multiple-domains](https://coolify.io/docs/images/loadbalancer/multiple-domains.webp)

- **Плюсы**:
    - Легко понять и реализовать.
    - Легко управлять.
    - Легко масштабировать.
    - Требуется прокси на серверах — Coolify настраивает его автоматически, но это еще один компонент, который потенциально может выйти из строя.
    - "Дешевле" (так как один сервер может хостить несколько приложений).
- **Минусы**:
    - Небольшая потеря производительности из-за дополнительного прокси на каждом сервере (незаметно в большинстве случаев).
    - Нужны более мощные серверы для работы сразу нескольких приложений.
    - Проверки состояния (healthchecks) могут быть доступны только для сервера целиком, а не для каждого приложения в отдельности.

## Docker Swarm

Coolify поддерживает Docker Swarm (экспериментально). Подробнее об этом можно прочитать [здесь](/knowledge-base/docker/swarm).

## Kubernetes (в планах)

Это только в планах, функции еще нет в дорожной карте (roadmap), поэтому сроков нет.
Coolify со временем будет поддерживать Kubernetes. Это позволит использовать всю мощь K8s вместе с удобным веб-интерфейсом для управления приложениями.
